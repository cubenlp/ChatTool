## 1. 当前种子想法

### 模块范式标准化，支持并发扩展开发
- 场景：模块越来越多，团队/多线程开发时易出现风格不一致和耦合上升。
- 目标：给每个模块约定统一范式，降低协作摩擦，支持并发发散开发。
- 方案：
  - 约定标准骨架：`core`（能力实现）/`cli`（命令入口）/`server`（可选服务化）/`skill|mcp`（Agent 接入）。
  - 约定输入输出规范：参数命名、错误码、日志字段、结果结构。
  - 约定最小测试集合：单测 + smoke（CLI）+ 可选集成测试。
- 成本/风险：前期会有重构成本，需要迁移历史模块。

### 延迟加载（Lazy Import）降低启动耗时
- 场景：功能增多后，CLI 或服务启动时会加载大量非必需依赖。
- 目标：减少冷启动时间，避免“没用到的模块也被 import”。
- 方案：
  - 把重依赖按需放入函数内部导入（命令触发时再加载）。
  - 对可选依赖使用明确报错提示（缺依赖时给安装建议）。
  - 建立启动耗时基线，对比改造前后数据。
- 成本/风险：滥用 lazy import 会让错误暴露延后，需要配合测试覆盖。

### 发散推进 + 阶段收敛重构（局部/整体）
- 场景：早期需要高速迭代，若过早重构会拖慢进度；但长期不收敛会积累技术债。
- 目标：在保证开发速度的同时，周期性规范化代码与框架。
- 方案：
  - 日常阶段以“功能推进”为主，允许受控范围内的实现差异。
  - 设定固定“收敛窗口”（例如每 2-4 周）做局部重构、接口统一和文档补齐。
  - 必要时设“守门角色”（或轮值）负责可用性、规范一致性和回归风险控制。
  - 大版本前进行一次“整体审视”，决定是否框架级调整。
- 成本/风险：若收敛窗口执行不坚决，债务会继续累积。

### 四层能力地图（Code -> CLI -> Server -> Skill/MCP）
- 场景：统一不同模块的演进路径。
- 目标：避免“功能能跑但不可维护/不可被 Agent 使用”。
- 方案：每个模块都画成熟度卡片并补短板。
- 成本/风险：前期会增加文档和测试工作量。

### DNS/Cert 作为样板模块
- 场景：已有代码基础和多种调用形态。
- 目标：产出可复制模板，带动其他模块升级。
- 方案：补齐端到端验收、错误码规范、Skill 路由文档。
- 成本/风险：要处理云厂商 API 变动带来的回归问题。

### Chat 能力产品化入口
- 场景：SDK 强，CLI/服务化入口相对弱。
- 目标：降低非 Python 用户使用门槛。
- 方案：评估新增 `chattool chat` 命令或轻量服务化接口。
- 成本/风险：功能边界容易与 Lark/MCP 重叠。
