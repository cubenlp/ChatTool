# ChatTool Roadmap（方向版）

更新时间：2026-03-08  
说明：只定义优先顺序与投入级别，不绑定具体时间线。

## 1. Roadmap 使用方式

- 关注点：先做什么、后做什么、为什么这个顺序。
- 不做：具体日期承诺与甘特式计划。
- 配套：每个方向标注工作量与难度，便于动态排期。

工作量分级：
- `S`：1-3 天量级
- `M`：3-10 天量级
- `L`：2-4 周量级
- `XL`：跨月、跨板块协同

难度分级：
- `低`：局部改动，依赖少
- `中`：跨模块联动，有回归风险
- `高`：架构/流程变更，需反复验证

## 2. 执行顺序（由近到远）

### R1 统一基础范式与最小护栏
- 做什么：
  - 明确各板块边界（tools/cli/mcp/setup/utils/application/tests）。
  - 固化最小规范（命名、错误、日志、最小测试、最小文档）。
  - 建立“发散推进 + 收敛窗口”执行机制。
- 为什么先做：这是后续并发扩展的底座，不先统一会放大返工。
- 工作量：`M`
- 难度：`中`

### R2 建立工具能力样板（tools 主线）
- 做什么：
  - 选一个高价值模块（建议 DNS/Cert）做标准样板。
  - 按 `Code -> CLI -> Server -> Skill/MCP` 补齐关键缺口。
  - 沉淀可复用模板与交付清单。
- 为什么第二步：先跑通一个样板，后续复制成本最低。
- 工作量：`L`
- 难度：`中-高`

### R3 优化交互与可发现性（cli/mcp/skills）
- 做什么：
  - 统一 CLI 子命令体验与错误提示。
  - 梳理 MCP 能力暴露与标签管理。
  - 补齐/统一 Skill 文档（中英双语优先核心模块）。
- 为什么第三步：能力已有后，需要让用户和 Agent “找得到、用得对”。
- 工作量：`M-L`
- 难度：`中`

### R4 强化交付链路（setup/docker/docs）
- 做什么：
  - 安装/部署路径收敛（本地、docker、setup）。
  - 文档按场景组织（上手、运维、集成）。
  - 排障路径标准化（依赖缺失、配置错误、权限问题）。
- 为什么第四步：降低使用门槛，减少支持成本。
- 工作量：`M`
- 难度：`中`

### R5 性能与稳定性治理（跨板块）
- 做什么：
  - 推进 lazy import，降低冷启动与非必要加载。
  - 形成核心回归集合（单测 + smoke + 关键集成）。
  - 建立发布前质量门槛与回归检查。
- 为什么第五步：在能力扩展后，集中处理性能和质量债务收益更大。
- 工作量：`L`
- 难度：`高`

### R6 场景化应用与生态扩展（application + provider）
- 做什么：
  - 把高频场景打包成可复用 workflow（如 DNS+Cert、Lark+AI、KB）。
  - 评估新增 provider 与平台接入优先级。
  - 形成“核心能力 vs 实验能力”分层策略。
- 为什么放后面：在底层机制稳定后，场景扩展才可持续。
- 工作量：`L-XL`
- 难度：`中-高`

## 3. 持续收敛触发条件

出现以下信号时，优先进入收敛而非继续发散：

- 新增功能明显拖慢已有功能开发。
- 同类模块接口差异过大，复用困难。
- 回归失败频发或排障成本快速上升。
- 文档与实际行为明显不一致。
- 启动耗时/资源占用持续恶化。

## 4. 决策优先级（冲突时）

1. 保证可用性和稳定性  
2. 保证扩展一致性（范式与注册层）  
3. 再追求功能广度和生态覆盖
