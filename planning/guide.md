# ChatTool 开发指引 Guide（Draft）

更新时间：2026-03-08  
状态：Draft v0（先可用，后完善）

## 1. 目标与原则（整体视角）

本指引用于统一 ChatTool 的开发方式，兼顾两件事：

- 保持开发速度：先把能力做出来，支持发散并发开发。
- 控制长期成本：通过阶段性收敛，把能力沉淀成可维护资产。

同时覆盖两个层面：

- 项目层：板块协同、阶段目标、整体演进与发散探索。
- 模块层：具体能力（尤其 `tools`）如何高效开发与标准化落地。

执行原则：

- 推进 + 重构双轨：日常迭代以功能推进为主；按触发条件进入收敛窗口做规范化与重构。
- 最小规范护栏：不过早做大重构，但必须满足最小结构、最小测试、最小文档。
- 板块分治：先区分板块职责，再选择对应演进路径，不做“一刀切”。

## 2. 项目整体板块地图（不是只有 tools）

当前项目可抽象为 6 个协同板块：

- 能力板块：`tools/`（核心能力生产）。
- 交互板块：`cli/`（本地入口）、`mcp/`（协议入口）、`skills/`（Agent 说明）。
- 基建板块：`utils/`（通用基础能力）。
- 交付板块：`setup/`、`docker/`、文档站（安装与运行分发）。
- 应用板块：`application/`（如 KB/Zulip 的场景化应用）。
- 质量板块：`tests/`、CI、发布流程。

目标不是让所有板块都“工具化”，而是让它们形成可演进的闭环：
`能力实现 -> 对外暴露 -> 安装交付 -> 场景应用 -> 质量回流`。

## 3. 仓库分层（推荐心智模型）

- `tools/`：核心能力实现层（业务能力本体）。
- `cli/`（client 入口）：交互入口与命令封装层。
- `mcp/`：工具 MCP 暴露层与能力汇总管理层。
- `setup/`：安装与环境搭建层（本地、docker、nginx、skill 安装等）。
- `utils/`：通用基础能力层（网络请求、CLI 交互、基础函数等）。
- `skills/`：面向 Agent 的能力说明层（通常中英双语）。

板块关系说明：

- `tools` 是核心能力板块，但不是全部。
- `cli`、`mcp`、`skills` 是能力暴露与交互板块。
- `setup` 是环境与安装板块。
- `utils` 是基础设施板块。

## 4. 整体发展方向（发散空间）

除 `tools` 之外，项目层建议并行发散以下方向：

- 方向A 产品形态发散：SDK、CLI、服务化、Agent 协议接入的边界与组合方式。
- 方向B 生态发散：更多 provider（LLM/Image/DNS）、更多平台接入（IM、协作平台）。
- 方向C 分发发散：本地安装、docker、一键 setup、远程部署模板。
- 方向D 场景发散：个人自动化、团队运维、内容工作流、知识库应用。
- 方向E 治理发散：模块分层策略、核心/实验分级、技术债预算机制。

这些方向先发散探索，再进入收敛窗口做取舍，不要求一开始全部定型。

## 5. 工具模块标准范式

一个工具模块建议具备以下结构（可按复杂度裁剪）：

```text
tools/<tool_name>/
  core.py            # 必须：核心业务逻辑（可复用）
  client.py          # 建议：本地 CLI 封装
  serve/
    serve.py         # 可选：服务端封装（远程执行/隔离）
    client.py        # 可选：服务化场景下的客户端
  mcp.py             # 建议：MCP tool 适配
```

落地要求：

- `core/local client/mcp` 通常为必要形态。
- `serve` 在以下场景优先考虑：
  - 安全敏感（秘钥隔离、多租户、权限控制）。
  - 本地配置复杂（依赖环境重、跨机访问）。
  - 计算开销大（需异步任务或后台执行）。

## 6. 演进模型适用范围（先分板块）

`Code -> CLI -> Server -> Skill/MCP` 是“能力演进模型”，主要用于 `tools` 类模块。  
其他板块按其职责采用更轻量路径：

- `tools`：优先使用四层模型评估成熟度。
- `cli/mcp/skills`：作为 `tools` 的暴露层，强调一致性与可发现性。
- `setup`：强调安装路径、环境探测、失败修复指引，不强求 Server/Skill 形态。
- `utils`：强调稳定 API 与复用性，不直接追求四层覆盖。

## 7. 工具四层完成定义（DoD）

每个模块用四层定义成熟度：

1. `Code`：核心 API 可复用，输入输出清晰，异常可控。  
2. `CLI`：命令可脚本化，参数语义稳定，错误提示可执行。  
3. `Server`：支持远程调用、鉴权和状态管理（按需）。  
4. `Skill/MCP`：可被 Agent 发现、理解、规划调用（含 SKILL 文档）。

说明：不是所有模块都必须立刻到 Layer 4；核心模块优先达标，实验模块可分阶段推进。

## 8. 项目级开发节奏：发散与收敛

日常阶段（发散推进，项目层）：

- 目标：快速验证方向和能力，支持多人并发开发。
- 允许：受控实现差异、局部临时方案。
- 要求：不破坏最小护栏（见第 9 节）。

收敛窗口（阶段重构）：

- 触发条件：出现接口碎片化、回归频发、文档失真、启动性能明显下降等信号。
- 目标：统一接口、清理债务、补齐测试/文档、审视架构与板块边界。
- 范围：优先局部重构；必要时做整体框架调整。

## 9. 最小规范护栏（必须满足）

- 命名规范：命令、参数、配置项风格一致（同类模块尽量复用同名参数）。
- 错误规范：可预期错误给明确提示，包含“如何修复”建议。
- 日志规范：保留最小结构化字段（模块、动作、结果、错误原因）。
- 测试规范：至少包含核心单测 + 关键 CLI smoke 测试。
- 文档规范：至少提供一个最小可跑示例和失败排查条目。

## 10. 路由与注册表模式（推荐）

为可扩展模块提供统一路由层/注册表，便于横向扩展：

- 适用：`chatenv` 配置类型扩展、image provider 扩展、llm 路由扩展、setup 子能力扩展。
- 目标：新增能力时尽量“增量注册”，避免改动大量入口代码。
- 好处：CLI 打包与交互逻辑可复用；新模块可从路由层快速验证。

## 11. 性能策略：Lazy Import

目标：随着能力增多，降低 CLI/服务冷启动耗时。

建议：

- 把重依赖放到具体命令函数内按需导入。
- 可选依赖缺失时，提供明确安装提示（如 extras 名称）。
- 建立启动耗时基线，持续比较改造前后数据。

注意：

- 避免无序 lazy import 导致错误暴露过晚。
- 必须用测试覆盖关键路径，确保首次调用时行为稳定。

## 12. 新模块交付清单（Checklist，按板块适配）

- `tools` 模块：`core` + 单测；推荐补 `CLI`/`MCP`/`skills`。
- `setup` 模块：安装命令可复现，失败提示可执行，文档有环境前置条件。
- `utils` 模块：接口稳定、复用清晰、被调用方有覆盖测试。
- 通用要求：至少一个最小示例 + 一条可复现 smoke 路径。

## 13. 整体发展发散清单（持续版本）

- 是否形成“核心产品线 + 实验产品线”的双轨发布机制。
- 是否将 `application` 从示例升级为独立可复用解决方案包。
- 是否建立“能力目录页”，支持按场景检索 CLI/MCP/Skill 路径。
- 是否为高频场景提供一键工作流（如 DNS+Cert、Lark+AI、KB+检索）。
- 是否引入板块级 owner 或轮值机制，提升协作与收敛效率。

## 14. 当前优先方向（用于近期执行）

- 先以 DNS/Cert 做 `tools` 四层样板，沉淀模板。
- 推进模块范式标准化，支持并发扩展。
- 在不影响功能迭代的前提下逐步引入 lazy import 优化。
- 用收敛窗口机制在必要时清理技术债并审视框架演进。
