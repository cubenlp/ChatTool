# ChatTool Planning (Living Document)

更新时间：2026-03-08
状态：v0（可讨论草案）

## 1. 项目现状快照

- 代码形态：Python 多能力工具箱，核心由 `chattool` 包 + CLI + MCP server 构成。
- 已有能力：Chat API 封装、Lark 机器人、DNS/DDNS/证书、图片生成、网络扫描、Zulip/KB。
- 交付形态：PyPI 包 + 命令行工具 + 文档站（mkdocs）+ 较完整测试目录。
- 当前风险：能力面较广，模块间一致性、文档同步、稳定性回归成本会持续上升。

## 2. 规划目标（先对齐）

- 目标A：把 ChatTool 打造成“一个入口，稳定调用多种自动化能力”的工程化工具箱。
- 目标B：减少新用户上手路径（从安装到第一个成功命令控制在 10 分钟内）。
- 目标C：把高频能力做到可观测、可测试、可维护，避免功能扩张导致维护失控。

## 3. 规划原则

- 先稳后广：先加固已有核心链路，再扩展新能力。
- CLI 与 SDK 同步演进：对外命令体验和 Python API 一起设计。
- 文档即产品：每个关键能力必须有“最小可跑”示例和失败排查。
- 默认可测试：新能力至少有单测；对外集成路径要有集成测试或模拟测试。
- 场景分层演进：每个能力按 `Code -> CLI -> Server -> Skill/MCP` 的四层路径建设。
- 节奏双轨：日常“推进功能优先”，按周期进入“收敛重构窗口”（局部优先，必要时整体调整）。
- 最小护栏：不过早重构，但保留最小规范（命名、错误码、日志字段、测试底线）防止失控。

## 4. 核心方法论（来自 code_tool_skill 模块）

统一把每个能力模块按四层评估和推进：

- Layer 1 `Code`：是否有稳定可复用的 Python API。
- Layer 2 `CLI`：是否有一致、可脚本化的命令行入口。
- Layer 3 `Server`：是否支持远程调用、鉴权、异步任务或多租户隔离。
- Layer 4 `Skill/MCP`：是否可被 Agent 发现、理解、规划调用。

这套模型的价值：

- 给“新增功能”一个明确完成定义，不再只看“功能能跑”。
- 让“面向人”和“面向智能体”的能力共用同一条演进主线。
- 有利于统一文档、测试和发布验收口径。

## 6. 建议优先级（P0/P1/P2）

## P0（近期必须完成）

- 配置体验统一：`chatenv`、`.env`、运行时读取逻辑和报错文案统一。
- CLI 可用性统一：子命令参数风格、错误码、help 文案、日志级别一致。
- 模块范式草案：定义模块标准骨架与输入输出约定，支持并发扩展开发。
- 稳定性基线：梳理核心回归用例（chat / dns / lark / mcp），明确 CI 必跑集合。
- 文档主路径：整理“3 条黄金路径”
  - 新手：安装 -> 配置 -> Chat 首次调用
  - 运维：DDNS + cert-update
  - 集成：MCP server 接入客户端

## P1（中期增强）

- 模块边界重构：降低 `cli`、`tools`、`application` 间耦合，增强可替换性。
- 启动性能优化：对重依赖进行 lazy import 改造，减少冷启动耗时。
- 观测能力：关键命令增加结构化日志和最小诊断信息。
- 发布流程：版本规范、更新日志模板、兼容性说明模板化。

## P2（探索方向）

- 插件化能力目录（可选安装、可选加载），与 Skill 路由一一映射。
- 远程配置或 profile 同步能力。
- 面向团队的场景化“解决方案包”（如飞书自动化、家庭网络运维包）。

## 7. 里程碑草案

## M1（1-2 周）

- 输出统一 CLI/配置规范文档。
- 补齐黄金路径文档与 smoke 测试脚本。
- 建立核心能力回归清单并接入 CI。
- 先选 1 个模块完成四层样板（建议 DNS/Cert），形成“标准模板”。

## M2（3-5 周）

- 完成 P0 收敛，修复高频失败点。
- 完成一次以“新用户 10 分钟上手”为标准的体验回归。
- 进行第一次“收敛窗口”执行复盘：确认哪些规范有效、哪些限制需要放宽。

## M3（6-8 周）

- 推进 P1 的模块边界优化与发布流程规范化。
- 形成下一阶段（P2）是否投入的决策依据（使用数据 + 维护成本）。
- 至少 2 个高频模块达到 Layer 4（Skill/MCP 完整可用）。

## 8. 度量指标（先定义再迭代）

- 上手成功率：从安装到首个成功命令的成功比例。
- 核心命令成功率：chat / dns / lark / mcp 在 CI+手测中的通过率。
- 文档有效性：按文档执行一次通过率。
- 回归效率：每次发布前的回归耗时与失败定位时间。
- Agent 可调用率：给定任务时，工具被正确选择并成功执行的比例。

## 9. 风险与应对

- 风险：外部 API 变动频繁（OpenAI、云厂商、Lark）。
- 应对：抽象 provider 层 + 失败降级 + 清晰错误提示。

- 风险：功能分散导致维护疲劳。
- 应对：按使用频率设核心/实验分层，实验能力不阻塞主线发布。

- 风险：文档滞后。
- 应对：把文档更新加入 PR 检查清单。
