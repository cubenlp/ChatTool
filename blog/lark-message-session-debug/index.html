    飞书消息与会话调试：先把要摸索的内容定清楚¶

    这篇先不讲“功能已经做好了什么”，而是先把一个新专题的探索范围缩清楚：

    • 我们到底要调什么
    • 哪些问题属于“消息调试”
    • 哪些问题属于“会话调试”
    • 第一轮应该先打通哪条最小链路

    如果这些边界不先定清楚，后面很容易一头扎进 SDK 细节、事件枚举和平台对象里，最后做成一堆零散命令，却没有形成稳定的调试工作流。

    先说结论：这次专题不是要重做一套大而全飞书 CLI¶

    ChatTool 现在对飞书能力的定位已经很明确：

    • 官方 OpenAPI 广覆盖能力，默认优先走 lark-cli
    • ChatTool 自己只保留最短调试链路

    当前仓库里，这条最短链路主要就是三件事：

    • chattool lark info
    • chattool lark send
    • chattool lark chat

    所以这次“消息与会话调试”专题，不应该演变成“继续往 chattool lark 里补全量命令面”。

    更合理的目标是：

    把和“发消息、收消息、识别会话、定位问题、复现问题”直接相关的调试路径做扎实。

    为什么要把“消息”和“会话”分开看¶

    这两个词经常被混在一起，但它们对应的问题面并不一样。

    消息调试¶

    消息调试关心的是一条具体消息有没有走通：

    • 发出去没有
    • 发给谁了
    • 用的什么 receive_id_type
    • 返回了什么 message_id
    • 是新发消息，还是 reply
    • 平台实际展示出来的内容是不是预期的

    这部分偏向“单次动作的正确性”。

    会话调试¶

    会话调试关心的是上下文有没有被正确识别和持续使用：

    • 这条消息属于哪个 chat_id / open_id / 会话对象
    • 单聊和群聊的路由是否一致
    • 回复时是否回到了原来的会话
    • 多轮对话状态有没有串用户
    • 清空历史、截断历史、切换 system prompt 后行为是否符合预期

    这部分偏向“连续交互中的状态正确性”。

    如果不把这两层拆开，最后就会出现一种假象：

    • 明明消息能发
    • 但用户还是觉得“机器人不对”

    实际上问题很可能根本不在“发消息”，而在“会话绑定、上下文归属、历史清理、回复位置”。

    这次探索的最小闭环¶

    第一轮不要追求复杂功能，先定义一个最小闭环：

    1. 能验证当前凭证和基本配置是活的
    2. 能向指定对象发一条最简单的文本消息
    3. 能知道这条消息落到了哪个会话
    4. 能在同一个会话里继续发第二条消息
    5. 能在本地复现一份最小对话状态
    6. 出问题时，能拿到足够的信息判断问题落在哪一层

    换句话说，我们先要的不是“能力多”，而是“定位快”。

    第一批要摸索的问题¶

    下面这些问题，适合直接作为本专题第一轮探索清单。

    1. 消息标识和会话标识要怎么区分¶

    这是最基础的一层。

    至少要把这些概念彻底理顺：

    • message_id
    • chat_id
    • open_id
    • user_id
    • union_id
    • receive_id_type
    • 单聊 / 群聊 / 话题回复之间的关系

    如果概念层不清楚，CLI 参数和调试输出一定会混乱。

    我们需要的不是再解释一遍平台术语，而是回答这类实际问题：

    • 我手里拿到一个 ID，它到底能不能直接发消息
    • 当前命令到底是“给用户发”，还是“给会话发”
    • reply 依赖的是 message_id 还是 chat_id
    • 一个报错到底是“目标错了”还是“权限不对”

    2. 最短发送链路里需要保留哪些返回信息¶

    现在最短发送路径已经有了，但从调试角度看，还需要确认返回信息是否够用。

    要重点看：

    • 发送成功后是否稳定返回 message_id
    • 是否能看到实际使用的 receive_id_type
    • 是否能在必要时打印 chat_id、接收目标和关键响应字段
    • 失败时错误信息是否足够区分“参数错误 / 权限错误 / 配置错误 / 对象不存在”

    如果一条发送命令只告诉你“失败了”，调试价值几乎为零。

    3. 收消息时我们最需要观察哪些字段¶

    接收侧是很多问题真正开始暴露的地方。

    这里要摸索的不是“把全量事件都 dump 出来”，而是找出最少但最关键的字段：

    • 发送者是谁
    • 当前会话是谁
    • 消息类型是什么
    • 原始文本是什么
    • 是否带引用 / 回复信息
    • 这条消息进入了哪个本地 handler

    如果后面要做 capture / replay，这批字段也会成为最小事件样本。

    4. 本地会话调试和真实飞书会话调试怎么接起来¶

    仓库里已经有两条线：

    • chattool lark chat
    • 适合本地调 prompt 和多轮状态
    • 飞书消息链路
    • 适合验证真实会话环境

    这次专题值得摸索的一个重点，是两条线怎么形成闭环：

    • 能不能先在本地把 prompt 和记忆轮数调顺
    • 再把同样的配置切到真实消息会话
    • 出问题时能不能快速判断是模型行为问题，还是平台会话问题

    如果这两条线始终割裂，排查会非常慢。

    5. 会话隔离策略要怎么观测¶

    多轮对话里最容易出事故的不是“没有记忆”，而是“记忆串了”。

    所以要明确会话隔离至少要回答：

    • 当前按什么 key 隔离
    • 单聊和群聊是否一致
    • 群里多个用户发言时，历史到底按谁隔离
    • /clear 这种动作清掉的是哪一层状态
    • max_history 截断后保留的到底是哪几轮

    这部分最好能形成明确的调试输出，而不是只靠读代码猜。

    6. 要不要支持消息 capture / replay¶

    这是一个很值得探索但不要太早做重的点。

    如果后面经常遇到“线上飞书里出现了一个奇怪对话，但本地复现不出来”，那 capture / replay 会非常有价值。

    它至少可以帮助回答：

    • 某次输入事件的原始样子是什么
    • 本地是否可以脱离飞书实时环境重放
    • 某个 bug 是平台事件差异，还是我们自己的路由问题

    但这里要先压住范围，第一轮不要直接做成大系统。 更现实的做法是先确定：

    • 最小 capture 文件长什么样
    • 最少需要保留哪些字段
    • replay 时只验证路由，还是连会话状态一起复现

    7. 哪些能力应该留在 ChatTool，哪些继续交给 lark-cli¶

    这是产品边界问题，不先定下来，后面实现会失控。

    我倾向于这样切：

    • ChatTool 负责
    • 最短消息调试
    • 最小会话调试
    • 本地 prompt / session 复现
    • 面向排查的 capture / replay
    • lark-cli 负责
    • 广覆盖 OpenAPI 能力
    • 文档、表格、日历、通讯录等标准资源操作
    • 大量平台对象的查询与管理

    这样分工是稳定的，不会重新把 ChatTool 拖回“大而全飞书封装”路线。

    这一轮先不做什么¶

    为了让专题收得住，下面这些内容先明确排除：

    • 不先扩大量新的飞书业务命令
    • 不先覆盖所有消息类型
    • 不先做复杂卡片编排
    • 不先做全量事件订阅面板
    • 不先做长期在线的重型消息代理

    先把最短文本消息与会话调试链路打实，再考虑增量扩展。

    可以直接落成任务的探索顺序¶

    如果要继续推进，这个顺序比较稳：

    1. 先梳理“消息 ID / 会话 ID / 接收目标”的术语和输出规范
    2. 重新检查 chattool lark send 的调试信息是否够用
    3. 为接收侧整理最小必看字段和日志格式
    4. 把本地 chattool lark chat 和真实飞书会话的调试流程串起来
    5. 评估最小 capture / replay 样本格式
    6. 最后再决定要不要引入新的 CLI 子命令或调试开关

    这个顺序的好处是，前面几步主要在澄清问题和稳定接口，不会太早陷入实现细节。

    我对这个专题的工作定义¶

    如果只用一句话概括，这次专题应该定义成：

    围绕“消息是否送达、会话是否正确归属、问题是否可复现”建立一条最小但稳定的调试工作流。

    后面所有新命令、新日志、新文档，都应该反过来接受这个定义的约束：

    • 它是不是让定位更快了
    • 它是不是让复现更容易了
    • 它是不是在扩大而不是收敛边界

    如果答案只是“又多了一个能调的平台接口”，那大概率就不是这个专题现在该做的事。
