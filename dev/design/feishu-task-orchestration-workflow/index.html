    飞书任务编排工作流设计¶

    目标：让任务执行尽量围绕**飞书会话**和**飞书文档**展开。会话负责接单、确认、追问、打断和收口；文档负责承载计划、过程记录、结果摘要和验收材料。

    背景¶

    这份设计稿需要基于当前已经验证过的现实边界，而不是基于旧版“大而全 chattool lark”设想继续扩写。

    当前已经比较清楚的事实是：

    • 飞书自动化默认应基于官方 lark-cli
    • ChatTool 自己只保留最短调试入口：
    • chattool lark info
    • chattool lark send
    • chattool lark chat
    • 文档正文、评论、权限、wiki 解析并不是同一条命令面：
    • 正文更适合走 docs
    • 评论与权限更适合走 drive
    • wiki 节点解析走 wiki
    • 在当前实践里，以下 bot 路径已经可用：
    • docs +create
    • docs +fetch
    • docs +update
    • drive +add-comment
    • drive file.comment.replys create
    • drive permission.members create
    • 但有些“发现型”动作不适合当一阶段默认面：
    • docs +search 更应视为 user-only 或至少不是 bot 默认链路

    因此，这份工作流设计不再把重点放在“飞书里什么都做”，而是收敛到一条更可靠的主线：

    会话作为控制面，文档作为状态面。

    设计目标¶

    • 用户主要在飞书会话里发任务、确认计划、追问进度、给 review。
    • 每个任务都有一份对应的飞书文档，作为计划、过程和结果的统一承载面。
    • 用户默认只面对一个外显角色，不需要理解内部是单智能体还是多智能体。
    • 关键阶段必须可见，但不把飞书会话刷成终端日志。
    • 工作流应优先依赖当前已经跑通过的 lark-cli 能力，而不是假设未来能力。
    • 最终交付应落到“一个会话 + 一份文档 + 必要评论 / 权限”这组最小对象上。

    非目标¶

    • 一阶段不做通用 DAG 调度器。
    • 一阶段不先做多维表格驱动的任务中心。
    • 一阶段不把飞书文档当作直接执行引擎。
    • 一阶段不要求所有发现、搜索、知识库浏览都自动化。
    • 一阶段不依赖 user login 才能跑通主链。

    一句话目标形态¶

    用户在飞书会话里发一个任务，例如：

    “帮我整理一版 agent-cli 的飞书工作流设计，并给我一份可 review 的文档。”

    系统默认走下面这条主线：

    1. 在会话里接单并复述任务理解
    2. 创建或绑定一份飞书文档
    3. 把计划写入文档
    4. 在会话里请求用户确认计划
    5. 执行过程中把关键进展追加到文档
    6. 在会话里回答“现在做到哪了”“下一步是什么”
    7. 需要 review 时，在文档中留评论或回复评论
    8. 需要交付时，补权限并把文档链接发回会话
    9. 最终在会话中给出简要收口，在文档中保留完整结果

    这条主线里，会话和文档的角色是不同的：

    • 会话
    • 短反馈、确认、提问、打断
    • 文档
    • 长内容、结构化计划、进度记录、结果沉淀、验收材料

    核心理念¶

    1. 会话是控制面，不是档案库¶

    会话里最适合放的是：

    • 任务接收
    • 计划确认
    • 阶段播报
    • 阻塞说明
    • 最终摘要

    会话里不适合长期堆积的是：

    • 完整计划正文
    • 长篇结果
    • 多轮修改痕迹
    • 可复用的验收材料

    所以会话应该尽量短，只承载“现在需要用户知道什么 / 决定什么”。

    2. 文档是状态面，不只是最终附件¶

    文档不应该只在任务结束时才出现。

    更合理的用法是：

    • 接单后尽快生成文档
    • 计划先写进文档
    • 执行中持续追加里程碑
    • 结果和验收摘要沉淀到同一文档
    • 用户 review 通过评论线程挂在文档上

    也就是说，文档是任务的长期工作面，而不是收尾时临时导出的附件。

    3. 评论是 review 面，不是主调度面¶

    文档评论在这个工作流里很重要，但它的职责应该收敛为：

    • 对文档某段内容提出 review
    • 针对评论做回复
    • 标记问题已处理 / 已解决

    评论非常适合 review，不适合承担整个任务调度逻辑。

    所以默认原则是：

    • 接单、确认、追问：走会话
    • 计划、过程、结果：走文档
    • review 反馈：走评论

    4. 权限和送达是主链的一部分¶

    很多设计稿只写“把文档生成出来”，但真实交付还差两步：

    • 用户是否有正确权限
    • 文档链接是否被送达到了正确会话

    因此一阶段主链里必须把这两件事纳入：

    • permission.members
    • 会话中的链接送达

    如果文档用户看不到，或者生成了却没回到会话，这条链就不算完成。

    当前可依赖的 lark-cli 能力边界¶

    已验证、适合作为一阶段默认面的能力¶

    • 文档创建：docs +create
    • 文档读取：docs +fetch
    • 文档正文更新：docs +update
    • 全文评论：drive +add-comment
    • 评论回复：drive file.comment.replys create
    • 协作者管理：drive permission.members create

    已知要谨慎使用的能力¶

    • docs +search
    • 更适合 user 身份，不应作为 bot 默认链路
    • wiki 节点解析
    • 可作为“已知 wiki 链接 -> 真实对象”的辅助步骤
    • 不应当成主流程默认必经环节

    设计含义¶

    这意味着一阶段最稳妥的主工作流不是：

    • 搜知识库
    • 扫很多文档
    • 自动发现上下文

    而是：

    • 已知任务上下文
    • 已知目标会话
    • 已知或新建目标文档
    • 然后沿文档主链推进

    推荐的顶层工作流¶

    顶层主链建议固定为：

    Conversation -> Doc Plan -> Execution -> Review -> Delivery -> Upgrade

    对应到用户可见阶段，可以收敛为：

    1. 已接单
    2. 待确认计划
    3. 执行中
    4. 待 review
    5. 已交付
    6. 已沉淀

    这里故意不再保留旧稿里的 Code -> Execute -> Feedback 那种偏内部实现的外显顺序。

    原因是对飞书端用户来说，更自然的问题是：

    • 任务收到了吗
    • 计划定了吗
    • 正在做吗
    • 现在能 review 吗
    • 最终交付了吗

    这才是会话和文档真正需要承载的状态。

    会话与文档的职责分工¶

    会话¶

    会话是用户和督工交互的唯一主入口，默认承担：

    • 发起任务
    • 明确范围
    • 确认计划
    • 询问进度
    • 打断 / 改方向
    • 接收最终摘要
    • 接收文档链接

    建议会话里的消息结构尽量稳定：

    • 接单消息
    • 计划确认消息
    • 阶段进展消息
    • 阻塞 / 需确认消息
    • 完成消息

    文档¶

    文档是任务的长期工作面，默认承载：

    • 任务背景
    • 计划
    • 执行记录
    • 结果摘要
    • 验收说明
    • 后续待办

    一份任务文档建议至少包含这些区块：

    # 任务标题

## Context
## Plan
## Progress
## Review Notes
## Delivery
## Next Step

    这样做的好处是：

    • 督工可以持续追加，不需要频繁改结构
    • 用户容易知道该看哪里
    • 评论落点清晰

    评论¶

    评论是附着在文档上的 review 面，建议只承担：

    • 指向某段计划或结果的 review
    • 回复处理动作
    • 标记已解决 / 待处理

    不建议把评论当作：

    • 主接单入口
    • 任务状态广播渠道
    • 复杂多角色调度协议

    权限¶

    权限是交付动作，不是附属动作。默认要考虑：

    • 谁可读
    • 谁可评论
    • 谁可管理
    • 是否需要给会话所在群可读

    一阶段默认对象模型¶

    为了让这条链可实现，一阶段对象先尽量少：

    1. Conversation¶

    表示任务主会话。

    最小字段：

    • conversation_id
    • conversation_type
    • starter
    • task_summary

    2. Task Doc¶

    表示任务对应文档。

    最小字段：

    • doc_token
    • doc_url
    • title
    • status

    3. Review Thread¶

    表示文档评论线程。

    最小字段：

    • comment_id
    • quote
    • status
    • last_reply_at

    4. Delivery State¶

    表示文档已经被送达和授权到什么程度。

    最小字段：

    • delivered_to_conversation
    • member_permissions_synced
    • group_read_enabled

    一阶段主链应该怎么跑¶

    Phase 1：接单¶

    用户在飞书会话里发起任务。

    系统要在会话里完成：

    • 复述任务理解
    • 给出一句短计划说明
    • 明确将创建或更新一份任务文档

    这一步不要求马上暴露内部是否会开多智能体。

    Phase 2：生成或绑定任务文档¶

    默认优先方案：

    • 若已有稳定任务文档，复用该文档
    • 否则创建一份新文档

    可依赖命令：

    • lark-cli docs +create
    • lark-cli docs +fetch

    Phase 3：写入计划¶

    计划应该先写文档，再回会话请求确认。

    会话里只放短消息，例如：

    • 文档已创建
    • 计划已写入
    • 请确认是否按此推进

    文档里放完整计划正文。

    可依赖命令：

    • lark-cli docs +update --mode append

    Phase 4：执行与进度记录¶

    执行中不需要把所有细节发回会话。

    默认只在两处更新：

    • 文档中追加关键进展
    • 会话中播报阶段变化或阻塞

    例如：

    • “计划已确认，开始执行”
    • “遇到权限问题，需要你确认 X”
    • “第一轮结果已写入文档，请 review”

    Phase 5：review¶

    当某一段计划、结果或交付需要 review 时：

    • 优先在文档对应位置留评论
    • 必要时在会话里同步一条短提醒

    可依赖命令：

    • lark-cli drive +add-comment
    • lark-cli drive file.comment.replys create

    这里的设计重点是：

    • review 应挂到文档内容上
    • 会话只负责提醒和追问

    Phase 6：交付¶

    交付不是“生成完文档”就结束，而是至少包括：

    • 补用户权限
    • 必要时补群可读
    • 把文档链接发回会话

    可依赖命令：

    • lark-cli drive permission.members create
    • 送达动作可由 chattool lark send 或消息侧能力承担

    Phase 7：升级沉淀¶

    任务结束后，再决定哪些内容需要升级为：

    • ChatTool 文档
    • skill
    • workflow 模板
    • CLI 改进

    这一步和 practice-make-perfact 是衔接关系，而不是前置依赖。

    为什么不再把多维表格放到一阶段核心¶

    旧稿里把“飞书线程 + 飞书文档 + 飞书多维表格”并列成三大载体，但从当前实践看，一阶段没必要同时把三者都做重。

    原因很直接：

    • 会话已经能承接接单、确认、追问
    • 文档已经能承接计划、结果、review
    • 多维表格更像跨任务总览面，而不是单任务闭环必需面

    所以更合理的优先级应是：

    1. 会话
    2. 文档
    3. 评论与权限
    4. 多维表格

    也就是说，多维表格可以保留为二阶段扩展，而不再是当前设计主轴。

    为什么不再把 cc-connect 放到默认底座¶

    旧稿里把 cc-connect 视为飞书入口底座，但这个表述现在偏重了。

    更准确的说法应该是：

    • 如果未来要做长期在线的会话代理，cc-connect 仍然可能有价值
    • 但这份工作流设计本身，不应该默认绑定在 cc-connect 上

    原因是这份文档现在关注的是：

    • 会话里应该说什么
    • 文档里应该写什么
    • 评论和权限如何进入主链

    这些问题首先是工作流问题，不是接入层实现问题。

    所以当前推荐表述应改成：

    • 接入层可以是 cc-connect
    • 也可以是其他会话代理
    • 但工作流本体应围绕“会话 + 文档”定义，而不是围绕某个桥接实现定义

    对 workflow 对象的最小要求¶

    如果后续要把这类流程沉淀成 workflow，对象本身至少要能约束：

    • 会话里何时发短消息
    • 文档里何时写完整内容
    • 何时需要评论进入 review
    • 何时必须补权限并送达
    • 完成后如何升级沉淀

    也就是说，workflow 至少要定义：

    • phase 顺序
    • 每个 phase 的输入输出
    • 会话模板
    • 文档模板
    • review 动作
    • delivery 动作

    如果只保留最小骨架，可以先约定：

    workflows/<name>/
├── WORKFLOW.md
├── manifest.toml
├── templates/
│   ├── conversation.md
│   └── doc.md
└── schemas/
    ├── plan.json
    ├── progress.json
    └── delivery.json

    当前最值得保留的结论¶

    当前阶段最值得保留的不是“飞书里什么都能编排”，而是下面这些更稳定的结论：

    1. 飞书工作流的主轴应该是**会话 + 文档**，而不是“消息桥接 + 各种平台对象并列”。
    2. 会话负责控制面：接单、确认、追问、打断、收口。
    3. 文档负责状态面：计划、进度、结果、验收。
    4. 评论负责 review，权限负责交付，不应再被当作边缘动作。
    5. 一阶段默认链路应严格依赖已经跑通过的 lark-cli 能力，而不是假设未来全覆盖。
    6. 多维表格和更复杂的调度底座都可以后置，不必成为当前设计前提。

    因此，这份设计稿的当前推荐方向是：

    先把“会话接单 + 文档承载 + 评论 review + 权限交付”这条最小闭环做扎实，再考虑更重的在线代理、多任务总览和复杂编排层。
