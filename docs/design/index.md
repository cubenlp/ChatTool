# Design

本文档用于收敛 ChatTool 的板块职责与开发流程，目标是保持快速迭代的同时降低维护成本。

## 设计目标

- 保持能力快速生长，不阻塞新增工具开发。
- 明确板块边界，避免逻辑在 CLI、Tool、Serve 之间漂移。
- 统一 CLI 交互行为与配置读取规则。
- 让文档结构直接映射代码结构，降低新人理解成本。

## 板块职责

### 1) `llm/`

- 负责模型路由、基础对话接口与流式接口。
- 可扩展模型编排能力，但不承载具体业务 Tool 逻辑。
- 对外提供稳定的 Python API，供 tools/serve/client 复用。

### 2) `config/`（env 管理）

- 统一管理环境变量与默认值加载。
- 统一解析顺序：`specific 参数 > default config file > environment variable`。
- 以模块化配置对象组织，支持按功能扩展。

### 3) `tools/`（核心能力）

- 每个工具目录负责该能力的完整实现。
- 同一工具应提供三类接入：
  - 代码接入（Python API）
  - CLI 接入（命令实现）
  - MCP 接入（Tool 暴露）
- CLI 是主能力形态，交互体验通过 `utils/tui.py` 统一增强。
- 当前散落在 `cli/` 的工具逻辑应逐步回收至对应 `tools/<name>/`。

### 4) `skills/`

- 面向 Agent 的能力出口。
- 将工具能力按场景组织为可复用技能，而非重复实现业务逻辑。

### 5) `setup/`

- 负责环境安装与配置初始化。
- 涉及 sudo 的操作默认打印脚本由用户执行。
- 可提供 `--sudo` 显式执行路径，但默认保持安全可审阅。

### 6) `docker/`

- 维护常用 docker compose 模板与 `.env` 生成逻辑。
- 对外提供统一配置入口，避免模板与代码双轨漂移。

### 7) `serve/`

- 承载云端服务能力，适用于算力不足或密钥敏感场景。
- 目标是独立服务实现，避免服务逻辑长期驻留 `cli/serve`。
- 与 `client/` 形成清晰的 server-client 职责拆分。

### 8) `client/`

- 提供统一的可执行命令入口。
- 负责本地工具调用与远程服务调用编排。
- 只做接入与编排，不承载核心业务实现。

## CLI 统一原则

- 必要参数不完整时，自动触发 interactive。
- `-i` 强制 interactive，`-I` 强制非 interactive；参数不全时抛错。
- 参数提示读取默认值并显示脱敏内容（密钥必须 mask）。
- 贯彻最小 import，避免 CLI 启动性能退化。

## 文档组织

- `docs/index.md`：架构总览与快速入口。
- `docs/env/`：配置与环境管理规范。
- `docs/tools/`：工具使用与能力说明。
- `docs/design/`：设计决策、边界规则、演进记录。
- `docs/blog/`：实践文章与经验沉淀。
- `setup` 与 `docker` 文档可放在同一组导航下统一讲解环境部署。

## 规范文档

- [目录职责说明](directory-responsibilities.md)：目录边界、依赖方向与归位规则。

## 暂缓处理

- `application/`：当前设计尚未定型，暂不纳入本轮收敛范围。
- 现有 `serve/` 浏览器相关代码：保留现状，后续按服务化方案单独清理。

## 当前执行顺序（无里程碑）

- 先统一规范源：以 Design + AGENTS + docs/index 为唯一有效规则，移除冲突描述。
- 再统一 CLI 规则：补齐全局 `-i/-I` 与缺参交互策略。
- 再做目录归位：将工具 CLI/MCP 逐步回收到 `tools/<name>/`。
- 最后做 serve/client 解耦：服务实现进入 `serve/`，client 仅做调用编排。
