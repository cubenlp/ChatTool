# 飞书任务编排工作流设计

> 目标：通过**一个飞书入口**承接任务、展示整体计划、推进单智能体或多智能体执行，并把进度、结果、验收概要统一收敛给用户；用户主要只负责发 `task` 和做 review。

## 背景

当前我们已经有两类相关能力：

- `practice-make-perfact`
  - 更偏任务完成后的沉淀习惯
  - 强调“先做事，再把能力整理成工具、skill、文档”
- `cc-connect`
  - 更偏把本地 Agent 接进 IM
  - 强调“把终端能力搬到飞书、Telegram 等聊天入口”

但这两者都还不是我们想要的那种任务编排工作流。

我们真正想要的是：

- 用户在飞书里只发一个任务
- 系统自动形成整体计划
- 系统按任务复杂度决定用单智能体还是多智能体分工
- 用户只面对一个“督工”入口，而不是面对一堆 bot
- 用户可以随时问进度、看结果、做 review
- 所有执行完成后，系统给出一份清晰的验收概要

这份设计文档描述的不是“开发者个人工作习惯”，而是一个面向飞书入口的任务执行产品工作流。

---

## 设计目标

- 一个飞书入口承接整个任务生命周期，不要求用户切换 CLI、GitHub、多个机器人窗口。
- 默认只暴露一个外显角色：督工。
- 督工统一负责：
  - 任务接收
  - 计划生成
  - 工作拆分
  - 状态汇报
  - 结果汇总
  - 验收收口
- 系统可根据复杂度自动选择：
  - 单智能体执行
  - 多智能体分工执行
- 用户主要只做三件事：
  - 发任务
  - 看计划 / 问进度
  - review / 验收
- 任务的关键阶段必须外显，不能把执行过程完全藏在黑盒里。
- 所有结果都应能回收到飞书线程和飞书文档里查看，不强依赖用户去看终端。

## 非目标

- 一阶段不做通用 DAG 工作流引擎。
- 一阶段不做任意复杂依赖图调度。
- 一阶段不要求用户显式配置多个 agent 角色。
- 一阶段不追求“完全无人值守”的自动开发闭环。
- 一阶段不替代代码托管平台的正式 Review / CI / PR 机制。
- 一阶段不先做 Web 控制台或公网任务中心。

---

## 一句话目标形态

用户在飞书里发出一句任务：

> “帮我给 ChatTool 设计一个飞书单入口任务编排 workflow，并给出文档落地方案。”

系统应完成下面的外显流程：

1. 督工接单
2. 生成整体计划，并写入飞书文档
3. 用户确认计划
4. 推进执行
5. 用户可随时在飞书线程里问进度、打断、展开
6. 汇总结果
7. 给出验收概要
8. 等待用户 review
9. 若有反馈，再进入下一轮
10. 一切顺利后，再进入复盘和升级沉淀

这里有两个一阶段前提：

- 计划记录优先放在飞书文档里
- 每个项目应有相对稳定的飞书文档工作面，后续任务可持续复用

## 核心理念

### 1. 用户只面对“任务”，不面对“编排细节”

用户关心的是：

- 任务有没有被理解
- 计划是否合理
- 现在做到哪了
- 最后交付了什么

用户不应该被迫理解：

- 当前拉起了几个 agent
- 哪个 agent 在读文件
- 哪个 agent 在跑命令
- 内部怎么路由 prompt

这些都应该由系统内部消化。

### 2. 督工是唯一外显 Agent

虽然内部可能有：

- Planner
- Coder
- Executor
- Reviewer
- Summarizer

但对用户来说，飞书里只应该看到一个统一入口：

- 督工

督工负责对外输出大概信息、配置和状态，不需要暴露过细执行细节。

### 3. 状态必须可见，但不必噪音过多

我们不是要把所有 token stream 和每个 shell 命令都刷到飞书里。

我们要的是：

- 关键阶段可见
- 当前阻塞可见
- 最终结果可见

而不是把飞书变成终端日志窗口。

后期可以考虑由 ChatTool 提供一些 Web view 来展示任务状态，例如在公网 server 启动服务，再把状态推过去；但这不属于一阶段目标。

### 4. 全过程尽量在飞书线程、飞书文档和飞书多维表格中完成

这套 workflow 的核心不是“再造一个控制台”，而是：

- 任务在飞书里发起
- 计划在飞书文档里展示
- 进度在飞书线程里追问
- 结果在飞书文档里汇总
- 任务总览在飞书多维表格中管理

因此，外部交互面应尽量收敛为：

- 飞书消息线程
- 飞书文档
- 飞书多维表格

三者的角色建议先分清：

- 飞书线程
  - 用来发任务、问进度、做 review
- 飞书文档
  - 用来承载计划、过程记录、结果摘要、验收概要
  - 后续可通过文档评论提出要求，再由机器人消费评论并继续推进
- 飞书多维表格
  - 用来承载任务列表、状态面板、负责人、阶段、优先级和总体视图

## 与 `practice-make-perfact` 的关系

这套工作流和 `practice-make-perfact` 有关联，但本质不同。

### `practice-make-perfact` 更像

- 任务后整理机制
- 开发习惯沉淀机制
- 能力归位规范

### 本文设想的 workflow 更像

- 任务运行时编排机制
- 飞书入口上的执行产品流程
- 面向“发任务 -> 看计划 -> 问进度 -> 做 review”的体验设计

它甚至不完全等同于通常意义上的 agent workflow。

某种程度上，它更像是一种文档模板和过程模板：  
通过模板和结构约束模型行为，而不是只靠自由提示词。

两者关系建议是：

- `practice-make-perfact` 作为任务完成后的升级阶段能力
- 本文 workflow 作为任务执行中的主工作流

也就是说：

> `practice-make-perfact` 不是主流程本身，而是本流程中的 `Upgrade` 阶段可挂接的一类后处理能力。

此外，我们通常让机器人在 `Playground` 下工作，ChatTool 集成核心能力，相当于共享大脑；任务结束后，再决定哪些经验需要更新回工具、文档、workflow 或 skill。

## 为什么不能只靠 skill

如果只是把经验沉淀成 skill，仍然有几个明显不足：

- skill 更偏提示词 / 路由说明，不天然承载任务状态
- skill 缺少明确阶段约束，容易“会做事但不按预期出结构化结果”
- skill 不天然包含 review 面、验收面、升级面
- skill 适合表达“如何做”，但不够适合表达“这类任务必须怎么推进”

因此我们需要一种比 skill 更强约束的对象，来描述：

- 任务类型
- 阶段定义
- 每阶段输入 / 输出
- review 约束
- 升级规则

本文把这个对象先称为：

- `workflow`

skill 仍然有价值，但更适合作为 workflow 的能力部件，而不是 workflow 本身。

## 候选实现路线

### 方案 A：复用 `cc-connect`

把 `cc-connect` 作为飞书入口和会话编排底座，在其上补：

- 督工层
- 任务状态层
- workflow 状态机
- 单 / 多智能体路由

#### 优点

- 已有飞书入口和 IM 通道
- 已有 agent 适配层
- 已有会话、relay、thread、群聊能力
- 已有把执行结果回写到聊天界面的路径

#### 缺点

- 当前更偏消息桥接，不是任务编排内核
- 缺少统一 `Task / Plan / Delivery` 状态对象
- 缺少更强约束的 workflow 模型

### 方案 B：以 `exec` 为中心做工作流执行器

把具体执行尽量收敛成标准化 step：

- 督工生成计划
- 各阶段都变成 `exec step`
- 每个 step 都有明确输入 / 输出
- 飞书层只负责显示状态和接收反馈

#### 优点

- 执行模型更统一
- 更容易做强约束 workflow
- 更容易建立阶段化输入 / 输出契约

#### 缺点

- 飞书入口、thread、群聊、会话能力仍要另补
- 如果完全绕开现有 `cc-connect`，集成成本偏高

### 方案 C：混合路线

推荐方向是：

- 飞书入口、thread、消息更新：复用 `cc-connect`
- 任务阶段、步骤执行、结果结构：引入 `workflow + exec step`

也就是：

> **入口层用 `cc-connect`，执行层用更标准化的 workflow step / exec 机制。**

#### 结论

- 推荐作为主方向

## 飞书平台约束

在继续往下推进前，需要先接受几个飞书侧的现实约束。

### 1. 飞书机器人之间不能相互 `@`

这意味着：

- 不能依赖“在群里 @另一个 bot”完成 bot-to-bot 路由
- 督工也不能通过原生 `@机器人` 机制去调度其他机器人

因此，多智能体编排不能建立在“机器人互相 mention”之上。

### 2. 飞书文档里不能 `@` 机器人

这意味着：

- 不能在飞书文档中通过 `@某个 bot` 触发执行
- 文档更适合作为状态、计划、结果和验收的承载层
- 文档不是任务触发通道，而是任务工作台和交付载体

### 3. 因此只能走“名称匹配 + 显式绑定”

既然不能靠 `@机器人`，编排里的目标寻址必须改成：

- 名称匹配
- 项目标识
- workflow 内显式绑定

也就是说，系统内部应该维护稳定的映射关系，例如：

- 督工名称 -> supervisor project
- 执行单元名称 -> project / workflow role
- 文档中的阶段名称 -> 内部 step / agent role

用户在飞书里看到和输入的，优先都应是稳定名称，而不是依赖平台原生 mention 能力。

## 核心工作流形态

顶层主链保持为：

```text
Task -> Plan -> Code -> Execute -> Feedback -> Upgrade
```

但一阶段不把它展开成复杂状态机，只保留用户可感知的几个阶段：

1. 已接单
2. 已出计划
3. 执行中
4. 待验收
5. 已完成
6. 已升级沉淀

这几个阶段已经足够支撑：

- 督工汇报
- 用户追问
- 结果验收
- 后续升级

---

## 我们当前要保留的核心设想

### 1. 一个飞书入口

- 用户只和督工对话
- 不要求用户切换多个 bot

### 2. 一个任务一个线程

- 每个任务优先绑定到一个飞书 thread
- 同一 thread 聚合计划、进度、结果、review

### 3. 一个项目一个主飞书文档

- 每个项目应有相对稳定的主文档
- 计划、过程记录、结果摘要、验收概要尽量沉淀到该文档体系
- 具体任务可以继续细分为子文档或文档章节

### 4. 一个任务一条结构化计划

- 计划不是只发在聊天里
- 计划应优先写入飞书文档
- 用户确认后再正式推进执行

计划确认这一步目前仍需要专门设计，但一阶段至少要有明确确认动作，而不是默认静默开跑。

### 5. 任务总览优先落到飞书多维表格

- 每个 task 一行
- 展示当前阶段、状态、项目、督工、优先级、更新时间
- 作为飞书入口之外的总览面板

### 6. 内部可单智能体，也可多智能体

- 小任务默认单智能体
- 可拆分任务使用多智能体
- 但这些内部差异默认不对用户外显

### 7. `Upgrade` 不只产出 skill

- 可以产出 skill
- 也可以产出 workflow
- 也可以产出 CLI、文档、design

---

## workflow 应该是什么

workflow 不是 skill 的别名，而是比 skill 更强约束的任务规范包。

它至少要能描述：

- 这类任务适用于什么场景
- 要经过哪些阶段
- 每一阶段要产出什么
- 最后如何汇总给用户
- 任务完成后如何升级沉淀

如果先只保留最小形态，workflow 至少应有：

```text
workflows/<name>/
├── WORKFLOW.md
├── WORKFLOW.zh.md
├── manifest.toml
├── schemas/
└── templates/
```

其中：

- `WORKFLOW.md`
  - 解释这类任务怎么推进
- `manifest.toml`
  - 规定阶段顺序、执行模式、升级策略
- `schemas/`
  - 规定计划、进度、结果的结构
- `templates/`
  - 规定飞书里展示给用户的稳定模板

## 最小 plan

当前阶段不直接做完整系统，而是先按最小计划推进。

### Phase 1：收敛文档形态

- 保留这份核心设计文档
- 不继续扩写底层实现细节
- 先明确：
  - 主目标
  - 推荐路线
  - 核心工作流
  - workflow 的最小定义

### Phase 2：确定底座

- 以 `cc-connect` 作为飞书入口和消息线程底座
- 不另起一个平行 IM 系统
- 后续执行层再逐步引入 `exec step`

### Phase 3：确定飞书载体分工

- 飞书线程：发任务、问进度、做 review
- 飞书文档：计划、过程、结果、验收
- 飞书多维表格：总览和状态面板

### Phase 4：定义最小 workflow 对象

- 先不做很多 workflow
- 先定义一个最小 workflow 规范
- 至少约束：
  - `Task`
  - `Plan`
  - `Feedback`
  - `Upgrade`

### Phase 5：先做一个督工最小闭环

先做到：

- 飞书里发任务
- 督工回计划
- 计划写入飞书文档
- 用户确认计划
- 督工能回答进度
- 最后能给结果摘要和验收概要

先不追求：

- 复杂 DAG
- 任意角色市场
- 过多可配置项
- Web 控制台

### Phase 6：再把常见任务升级成 workflow

等最小闭环稳定后，再把高频任务沉淀成标准 workflow，例如：

- 写设计文档
- 做调研
- 做 review
- 做发布检查

---

## 当前结论

当前更重要的不是把文档写成完整实现细则，而是先固定住下面几件事：

1. 一个飞书入口
2. 一个外显督工
3. 一条清晰工作流：`Task -> Plan -> Code -> Execute -> Feedback -> Upgrade`
4. 飞书线程、飞书文档、飞书多维表格三者分工
5. 一个比 skill 更强约束的 workflow 形态

在此基础上，推荐路线是：

> **复用 `cc-connect` 作为飞书底座，再逐步补上 workflow 和 exec step。**

这就是当前阶段最值得保留的核心设想和形态。
