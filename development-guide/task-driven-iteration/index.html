    任务驱动的技能与工具沉淀¶

    本文档记录在执行任务过程中如何持续沉淀工具与技能，并保证项目结构一致。

    基本流程¶

    1. 先完成任务，再规范化
      先完成实际需求，再做文档、命名与结构整理。
    2. 前期探索和方案形成阶段保持轻约束，不要让流程本身打断必要的代码阅读与尝试。

    3. 当实现方向明确后，再切换到严格的规范化与提交流程。

    4. 优先 CLI 交互
      能用 chattool CLI 完成的读取与导出，尽量使用 CLI，确保可复现与可审计。

    5. 探索期产生的临时脚本、草稿、导出结果，优先放到仓库外的独立工作目录，例如 ~/tmp/chattool/<task>/。

    6. 不要把 scratch、试验产物或一次性导出结果默认放进当前仓库，避免污染 git 历史和项目结构。

    7. CLI 测试文档先行
      CLI 测试必须先写对应测试线下的 .md，再实现对应 .py，确保步骤与预期可审阅。

    8. ChatTool 仓库长期维护两条 CLI 测试线：cli-tests/ 负责真实链路，mock-cli-tests/ 负责 mock 链路。
    9. 仓库根下 tests/ 仅作历史参考，不再作为新开发默认测试落点。
    10. 需要验证真实表现时，优先构造真实文件系统、真实仓库状态和真实配置，并写入 cli-tests/。

    11. 使用 mock、patch、monkeypatch、fake client / API 的 CLI 测试，统一写入 mock-cli-tests/。

    12. 能力归类

    13. 通用、可复用：补充到 src/（工具层、CLI、MCP）。

    14. 任务特定：补充到 skills/<name>/。

    15. 技能落地要求

    16. 必须包含 SKILL.md 与 SKILL.zh.md。
    17. SKILL.md 与 SKILL.zh.md 的 frontmatter 至少维护 name、description；version 仅在确有需要时作为可选元信息维护。
    18. 分阶段输出约定：0_ 表示原始或起点；1_ 起为加工内容。

    19. 若任务需要阶段化产物，目录层级保持简洁，优先使用仓库外目录下的 <频道>/<主题>/ 或 <任务>/<阶段>/。

    20. 任务后复盘

    21. 做一次代码 review，统一命名、结构与文档。
    22. 若需要把已完成任务统一整理成仓库规范，显式调用 skills/practice-make-perfact/ 作为后处理阶段，而不是在任务开始前套流程。
    23. 在这个后处理阶段，默认串联 skills/chattool-dev-review/ 做开发规范检查。
    24. 若任务继续进入合并后的版本 tag / 发布阶段，再切到 skills/chattool-release/，不要把正式发版动作混进开发 review。
    25. 若这次任务要进入正式发版，开发分支在 PR/MR 前就要更新 src/chattool/__init__.py 和对应 CHANGELOG.md 到目标版本；合并后只负责从主线打同版本的 vX.Y.Z tag，不要等到合并后再补改版本号。
    26. 正式发版的标准 tag 统一为 vX.Y.Z，且只能从已合并并同步到最新的主线创建。
    27. 如果 PyPI 已经存在该版本，禁止继续推同版本 tag 试图“重发”；必须先提交新的版本 bump 变更，再走下一次合并后发版。
    28. 更新导航（如 mkdocs.yml）与文档索引。
    29. 将任务推进到 PR/MR 阶段，而不是只停留在本地提交。
    30. GitHub 流程优先使用 chattool gh，包括 pr create、pr edit、pr view 等。
    31. 在声称“PR 可合并”或“CI 已通过”之前，必须先同步最新 base 分支，并验证一次真实合并态：git fetch origin <base> 后，查看 chattool gh pr view / pr checks 的 mergeable、mergeable_state，必要时在最新 base 上做临时 merge/rebase 演练并在该结果上跑最相关测试。

    结果目标¶

    • 每次任务都有产出：工具、技能或文档。
    • 项目结构不分叉，既能完成任务，也能积累长期能力。
